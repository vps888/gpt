## 3. 使用心得和问题

在家体验了两天，效果很不错。我的环境是：[MacMini + iPad + 自建VPS](https://github.com/vps888/vps2605)。

### 3.1 网络质量很重要，主要是稳定性和延迟问题影响体验

一开始我用的是新加坡节点，访问谷歌速度倒是挺快，但是ChatGPT的响应速度有点慢，主要表现为：

* 语音对话经常报错要重试；
* 命令下发后，过一会才有响应（体感慢，具体几秒钟没统计）；
* 锁屏后再次解锁，界面看不到结果，要好一会才能看到最新结果（体感更慢）。

一开始没想到是网络问题，就觉得访问谷歌很快就没问题。后来切换到自己搭建的**[三网优化（CN2 GIA / CMIN2 / 9929）的节点](https://github.com/vps888/vps2605)**后，延迟大幅降低，语音交互与代码流式输出变得极为丝滑。**进行远程 AI 编码时，链路的延迟（Ping 值）比单纯的带宽（吞吐量）更重要。**

### 3.2 虽说移动端可以看到过程和结果，但是实际效果如何查看？

移动端可以看到处理过程和处理结果，但是我们如何查看项目实际效果是个问题。通过这2天使用，我总结了两种方法：

* 利用 Tokens 换取图文反馈。也就是让Codex直接把运行结果通过文字或者截图的方式反馈。这种方法最简单，直接和Codex说就好了，它知道该怎么做。缺点就是要多花一些tokens，虽说Codex量大管饱，但是plus套餐的话，还不是那么富裕。
* 内网穿透（强烈推荐）。我们的桌面端（MacMini）通常是在内网中，家庭宽带很少提供公网IP。这种方法就是通过某个中间Server透传请求。比较适合http、ssh类请求，比如web类项目或者ssh登录等。本文暂不介绍，如果有需要可以评论留言，后续单独写一篇教程。

我就是内网穿透，通过域名绑定了Macmini本地端口，这样可以在公网上通过域名访问查看最终效果。延迟略有，但是可以忍受。
